A task is an operational work item that takes 1-7 days, contributes to a parent goal, and has a binary done/not-done outcome. It states *what* will be true when done plus the *concrete steps* to get there.

## TL;DR

- **Use for**: operational work, 1-7 days, binary done/not-done
- **Create**: `/vault-cli:create-task "<title>"`
- **Audit**: `/vault-cli:audit-task "<title>"`
- **Sections**: Summary ‚Üí Impact ‚Üí Success Criteria ‚Üí [DoD] ‚Üí **Out of Scope** ‚Üí Tasks ‚Üí Progress
- **Forcing functions**: ‚â• 1 parent goal linked; Out-of-Scope enumerated; criteria binary

## Goal

Produce a task page that is atomic, binary-completable, parent-goal-linked, and auditable by `task-auditor` without ambiguity.

## When to Write a Task

| Situation | Task? |
|-----------|------|
| Operational obligation (SHOULD do), 1-7 days | Yes |
| Concrete change with clear completion | Yes |
| Contributes to a parent goal's success criterion | Yes |
| Personal ambition, weeks-to-months effort | No ‚Äî write a goal |
| Multi-task umbrella outcome | No ‚Äî write a goal |
| Recurring routine (daily/weekly) | Yes, but use `recurring:` frontmatter |
| Code change inside an active goal | Yes; consider whether a dark-factory spec is needed instead |

## Creating a Task

Use the slash command:

```
/vault-cli:create-task "<title>"
```

The command invokes `task-creator`, which scaffolds a file in the configured vault's `tasks_dir` (typically `24 Tasks/`). The agent reads vault config via `vault-cli config list --output json` ‚Äî never hardcode paths.

## Task Structure

### Frontmatter

```yaml
---
status: todo
page_type: task
priority: 3                                      # 1 (highest) ‚Äì 3
category: <domain>                               # optional
goals:                                           # ‚â• 1 parent goal recommended
  - "[[Parent Goal]]"
assignee: bborbe                                 # who's holding the baton
task_identifier: <uuid>                          # generated by tooling
planned_date: 2026-MM-DD                         # optional
defer_date: 2026-MM-DD                           # optional, for snoozing
recurring: weekly                                # optional, for routine tasks
---
```

`status` valid values: `in_progress`, `todo`, `backlog`, `hold`, `completed`, `aborted`.

`assignee` semantics: empty (`""`) means **unclaimed inbox** (anyone with vault access can pick up); an agent name means the executor should spawn that agent; a human name means that human is currently doing the work.

### Required sections

In order:

1. `Tags: [[Task]]` (after frontmatter, before content separator)
2. **Summary** ‚Äî first paragraph after the `---` separator. Action-verb-led. 1-2 sentences.
3. `# Impact` ‚Äî why this matters; reference parent goal's success criteria explicitly when possible
4. `# Success Criteria` ‚Äî 2-4 binary checkbox outcomes (binary = done or not done)
5. `# Definition of Done` ‚Äî required for complex tasks (‚â• 4 success criteria, multi-phase, or ambiguous terms). Per-criterion verification: `**Automated**: ...`, `**Behavioral**: ...`
6. `# Out of Scope` ‚Äî recommended; explicit deferrals (parallels `# Non-goals` on goals)
7. `# Tasks` ‚Äî actionable subtask checkboxes (work-tracking; not parent-goal links). **Granularity rule:** session-sized work blocks, not CLI steps. Aim for 3-6 items. Collapse "write spec / audit / approve / generate prompts / precommit / open PR / merge" into one "ship the change through the pipeline" block. Each subtask should represent a meaningful unit a human can pick up, work on, and report back about ‚Äî not a single command.
8. `# Progress` ‚Äî log of work done, dated entries; lives at bottom

Optional: `# Related` ‚Äî sister tasks, docs, runbooks.

### Out of Scope ‚Äî the scope-creep guard

Adopted by parallel with goals' Non-goals convention. Explicit deferrals prevent task bloat:

**Good out-of-scope:**

```markdown
# Out of Scope

- Renaming the agent's CRD (separate concern)
- Migrating existing tasks in the vault (only new emissions affected)
- Schema changes in `bborbe/agent/lib`
```

**Bad out-of-scope:**

```markdown
# Out of Scope

- Future improvements
- Anything else
```

**Rules:**
- 2-5 items typical for tasks (less than goals; tasks are smaller)
- Each item is a *concrete* deferral or alternative
- Link a follow-up task if the deferred work is real

## Shipping Checklist ‚Äî required for shipping-class tasks

A **shipping-class task** ships a real-world artifact: PR/release, plugin update, agent definition, deploy, library publish, binary, deployment manifest.

Detect shipping flavor via signals: title or impact mentions `PR`, `release`, `tag`, `ship`, `deploy`, `publish`, `roll out`, `slash command`, `plugin`, `agent`, `library`; or subtasks reference a git repo / marketplace / registry.

When the task is shipping-class, the `# Tasks` section **must explicitly enumerate** these three items:

1. **Merge / land the change** ‚Äî PR merged, code on default branch
2. **Release fired** ‚Äî version tagged, artifact published. Don't trust `autoRelease: true` config alone; the tag must actually exist. Verify with `git tag --sort=-creatordate | head` or `gh release list`.
3. **End-to-end verification** ‚Äî the shipped artifact runs in its real environment (a real Claude Code session for a slash command, real cluster for a deploy, real install for a library). Audits and unit tests don't count; "deferred to first use" doesn't count.

**Anti-pattern:** ticking a verification subtask with body containing any of these dishonest-tick phrases (case-insensitive substring match):

- *"deferred to first use"*
- *"deferred ‚Äî will validate"*
- *"will check next session"*
- *"will verify on first use"*
- *"first deployment will test"*
- *"trust the audit"*
- *"trust CI"*
- *"trust the tests"*
- *"will validate later"*

The `task-auditor` flags any of these as MAJOR. Keep the subtask `[ ]` until real-environment execution evidence exists.

**Why the third one matters most:** audit skills (`/coding:audit-slash-command`, `/coding:audit-agent`) catch structural issues; bot reviewers catch some runtime bugs. Neither catches *"my new slash command isn't actually installed in this session yet"*. Only running the command end-to-end does.

## Scope Check

Before committing to a task, verify these signals:

- **Success criteria count 2-4** ‚Äî if ‚â• 5, complex enough to need DoD
- **Single mental model** ‚Äî if the task touches multiple unrelated domains/repos, split
- **Days-to-week effort** ‚Äî if multi-week, it's a goal masquerading as a task
- **Linked to ‚â• 1 parent goal** ‚Äî orphan tasks are scope creep by definition
- **Title is action-verb-led** ‚Äî "Add X", "Fix Y", "Migrate Z"; not "Stuff about X"

If 3+ smells fail ‚Üí split or promote to a goal.

## Preflight Checklist

Before approving:

- [ ] What outcome will be true when done (binary)?
- [ ] Does this contribute to a specific parent goal's success criterion?
- [ ] What's explicitly out of scope?
- [ ] Are success criteria binary (yes/no, not "improve X")?
- [ ] Is this 1-7 days of work, not weeks?
- [ ] Is the title action-verb-led?

## Audit

Always audit before committing publicly:

```
/vault-cli:audit-task "<task title or path>"
```

The auditor (`task-auditor` agent) checks structure, success-criteria binary-ness, parent-goal linkage, DoD presence for complex tasks, and Out-of-Scope clarity.

## Lifecycle

| Status | Meaning | Trigger to enter |
|--------|---------|------------------|
| `todo` | Defined, not started | Task file created with required sections filled |
| `in_progress` | Actively working | `/vault-cli:work-on-task` or `assignee` set + first subtask started |
| `hold` | Blocked | `blocked_by:` field populated, or operator sets manually |
| `completed` | All success criteria met | `/vault-cli:complete-task` ‚Äî checks every `# Success Criteria` checkbox is `[x]` |
| `aborted` | Abandoned without completion | Operator sets manually with reason in body |
| `backlog` | Not committed yet | Initial state before commitment |

Recurring tasks reset on `complete` rather than archive ‚Äî `recurring:` frontmatter drives the deferral cycle.
